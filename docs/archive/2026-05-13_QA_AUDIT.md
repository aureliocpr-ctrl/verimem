## 3. Top 10 untested critical paths

Ordinati per **rischio (sicurezza × probabilità di regressione × blast radius)**.

1. **`tools_extra.shell_run`** (LOC 215-236, 0% cov) — comando shell arbitrario sul sistema utente. Nessun test su path-traversal, command injection, timeout, output truncation. **Critical/security**.
2. **`tools_extra.fs_write_file` + `fs_read_file`** (LOC 56-103) — sandbox di filesystem con `_fs_roots()` permissivo (default = `$HOME`). Nessun test che validi che `HIPPO_FS_STRICT=1` funzioni davvero, nessun test su simlink escape, lunghezza path Windows (>260 char), file binari.
3. **`tools_extra.vision_describe` / webcam / desktop** (LOC 391-625) — capability gate `_enabled()` non testato; un bug nelle env-flag potrebbe lasciare aperte capability che dovrebbero essere off.
4. **`llm` provider matrix** (`AnthropicLLM`, `OpenAICompatLLM`, `OllamaLLM` — 31% cov) — tutti i 18 provider sono autodetected via env var, ma nessun test mock-respx valida la richiesta HTTP né la risposta parsing. Una regressione su `_supports_temperature` (Opus 4.7 fix) può rompere silenziosamente.
5. **`llm` fallback chain** (`scan_all_providers`, `resolve_model`, fallback retry) — logica complessa, zero test.
6. **`dashboard.chat_api` + `sleep_api`** (POST /api/chat, /api/sleep) — entry-point principale del dashboard. Nessun TestClient.
7. **`dashboard.feedback_api`, `skill_retire`, `skill_promote`** — mutazioni su skill library via HTTP, nessun test (race conditions, invalid IDs, autorizzazione assente).
8. **`mcp_server.call_tool` / `read_resource`** — contratto MCP completo non testato. Ogni handler (5 tools, 4 resources) può rompersi senza che nulla se ne accorga.
9. **`cli` end-to-end** — comandi `wake`, `sleep`, `code`, `chat`, `dashboard`, `mcp`, `metrics`, `reset`, `providers list/scan/models` — 0% cov. Una rottura del pipeline import equivale a `engram` rotto in produzione.
10. **`code.EngramCode` interactive loop** + slash command discovery + edit application — è il flagship product, untested.

Bonus risk paths (not top-10 ma watchlist):
- `wake.run` retry-after-critic path (linee 415-426) — non esercitato.
- `wake._try_compiled_macro` corner case quando macro fallisce (linee 477) — non testato.
- `sleep.cycle()` con LLM che lancia exception in mid-stage — robustezza non testata.
- `settings.save()` su disk full / permission denied — non testato.
- `embedding._model()` lazy-load race condition multi-thread — non testato.

## 4. Categorie di test mancanti

### P0 — bloccanti per v1.0

- **Unit test LLM client multi-provider con `respx`/`httpx_mock`**: una matrice (15+ providers) che mocka l'endpoint OpenAI-compatible / Anthropic / Ollama e verifica:
  - corretto routing autodetect → provider
  - corretto resolve_model() per ciascuna stage (executor/dreamer/critic)
  - corretto skip di `temperature` per Opus 4.7
  - corretto retry/fallback chain su 429/500
  - corretto parsing tool_calls native (Anthropic vs OpenAI format)
- **Integration test del wake-loop completo** (wake → tool → sleep → skill consolidation) con `MockLLM` scriptato e validator vero. Esiste solo per casi happy-path ridotti.
- **Contract test MCP server** (subprocess JSON-RPC) — verificare che la suite di tools/resources risponda al protocollo MCP.
- **CLI end-to-end con `subprocess.run`** + `typer.testing.CliRunner` per ogni sottocomando: import OK, --help OK, exit code corretto, side-effect minimo.
- **Dashboard FastAPI route test (TestClient)** per le 38 route: status code, schema risposta, validazione input (pydantic).
- **Sandbox/permission tests** per `tools_extra` — gate FS strict, gate webcam, gate computer-use.

### P1 — per maturità

- **Property-based tests con `hypothesis`** sui parser:
  - `editfmt.parse_edits` su input arbitrari (CRLF, fences nested, BOM, NULL bytes, CJK).
  - `wake.parse_react_step` su qualsiasi LLM output (tolerant parser).
  - `repomap` su file Python/JS/TS arbitrari.
- **Stress / concurrency test sleep consolidation**: 100 episodi paralleli + cycle che non fa starvation, no SQLite "database is locked".
- **Test di proprietà su Skill Bayesian fitness** — invarianti: `0 ≤ fitness ≤ 1`, monotonicity dopo n successi, prior decay.
- **Test su large-scale repomap**: repo da 5k file → repomap si genera in <X secondi, no OOM.
- **Test su `editfmt` per CJK / emoji / BOM / file binari** — solo CRLF è coperto oggi.

### P2 — qualità

- **Visual regression test** del dashboard (Playwright + percy/argos) — non bloccante ma utile.
- **Performance/latency benchmark** del wake-loop (`pytest-benchmark`) per detectare regressioni: target P95 wake step <2s con MockLLM.
- **Mutation testing** con `mutmut` o `cosmic-ray` sui moduli A-tier per validare la qualità degli assert.
- **Documentation coverage** (`pytest --doctest-modules`) — molti docstring contengono esempi che non sono mai eseguiti.

## 6. Issue rilevate durante l'audit

### I-1 — `sentence-transformers` non installato di default

**Sintomo**: 48/110 test falliscono con `ModuleNotFoundError: No module named 'sentence_transformers'` su un'installazione fresca seguendo `pyproject.toml [project.optional-dependencies] dev`.

**Causa**: `sentence-transformers` è in `[project] dependencies` ma il job CI o il dev fresh-install probabilmente usa solo `pip install -e ".[dev]"` senza forzare la deps core (testato qui in conda environment già "popolato"). Va aggiunto un esplicito `pip install -e .` step nel CI prima dei test, oppure il modulo `embedding.py` deve avere un fallback puro per i test (uso fixture deterministico hash-based).

**Suggerimento QA**: introdurre in `tests/conftest.py` un fixture `monkeypatch.setattr(embedding, "_model", lambda: _StubModel())` con uno stub deterministico, in modo che la test suite **non dipenda mai** da modelli HuggingFace (porta CI ~30s → ~5s e rende il test offline-safe).

### I-2 — Pytest config non emette warning su test collection vuota
Aggiungere `--strict-markers --strict-config -ra` agli `addopts`. Aggiungere `xfail_strict = true`.

### I-3 — Nessun `coverage` gate in CI
`pytest-cov` è in deps ma `pyproject.toml` non ha né `[tool.coverage.*]` né un job CI che lega `--cov-fail-under`. Una PR può abbassare la coverage senza nessuno se ne accorga.

### I-4 — Test del dashboard limitati al sub-router IDE
Tutti gli altri 28 endpoint sono fly-by-the-seat. Dopo il push di `feat(engram): full IDE — Monaco editor + WebSocket terminal`, l'unica testata. Le 38 route HTTP vanno coperte.

### I-5 — Nessun smoke test per il packaging
`pip install .` su un venv pulito + `engram --help` dovrebbe vivere come CI gate. Oggi non esiste.

### I-6 — `tests/test_compilation.py::test_wake_fast_path_bypasses_llm_when_macro_applies` dipende da `embedding`
Quando l'embedding model mancava, ha fallito a cascata. Vedi I-1: stubbare `embedding` in conftest sblocca la robustezza dei test sul dominio puro (compilation, wake, schema).

### I-7 — Nessun test di permission gate
`settings.UserSettings.perm_*` (filesystem, computer_use, webcam, shell, web, vision) sono campi pubblici ma nessun test verifica che il toggle effettivamente disabiliti la capability lato `tools_extra`. Security regression risk.

## 7. CI test matrix raccomandata

```yaml
# .github/workflows/test.yml
name: tests
on: [push, pull_request]
jobs:
  test:
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        python: ["3.10", "3.11", "3.12", "3.13"]
        exclude:
          # Risparmio CI: 3.13 solo su ubuntu, le altre triple
          - os: macos-latest
            python: "3.10"
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python }}
      - name: Install
        run: |
          pip install -U pip
          pip install -e ".[dev]"
      - name: Lint
        run: ruff check hippoagent tests
      - name: Tests + coverage gate
        run: |
          pytest --cov=hippoagent --cov-report=xml \
                 --cov-fail-under=85 \
                 -W error
      - name: Smoke install
        if: matrix.os == 'ubuntu-latest' && matrix.python == '3.12'
        run: |
          pip uninstall -y hippoagent
          pip install .
          engram --help
          hippo --help
      - uses: codecov/codecov-action@v4
        if: matrix.os == 'ubuntu-latest' && matrix.python == '3.12'
```

**Note**:
- `python: 3.10/3.11/3.12/3.13` — `pyproject.toml` dichiara `requires-python = ">=3.10"`.
- 3 OS — hippo si vuole cross-platform (Windows confermato, Linux ovvio, macOS per dev).
- Smoke install separato a coverage normale per non frammentare il gate.
- `-W error` per evitare deprecation warnings silenziosi.
- Aggiungere job dedicato `mcp-contract` con `mcp-cli`/`mcp inspector` per regression-test del server.

### Test stages suggeriti (pyramid)

- **Unit** (tier A/B): `pytest tests/unit/ -m "not slow"` — <30s
- **Integration** (full sleep cycle, MCP, dashboard): `pytest tests/integration/ -m integration` — <2 min
- **End-to-end** (subprocess CLI, packaging): `pytest tests/e2e/ -m e2e` — <5 min
- **Performance** (pytest-benchmark, opzionale): `pytest tests/perf/ --benchmark-only` — non bloccante

Marker da introdurre nel `pyproject.toml`:
```toml
markers = [
  "slow: marks tests that take >5s",
  "integration: cross-module tests requiring multiple components",
  "e2e: subprocess-based end-to-end tests",
  "security: permission/sandbox/path-traversal tests",
  "network: tests that require network (excluded by default)",
]
```

## 9. Conclusioni

**Stato attuale**: progetto in fase R&D maturo sul cuore cognitivo (memory, skill, sleep) con coverage A-tier eccellente (>87% in media), ma **interfaccia utente, integrazioni e capability sandbox sono untested**. Per arrivare a v1.0 production:

- Investire **3 settimane di QA full-time** per chiudere i gap P0 → 85% coverage.
- Adottare la CI matrix proposta (Windows/Linux/macOS × Py 3.10-3.13).
- Introdurre coverage gate hard a 85%, no exceptions.
- Stubbare `embedding` nei test per CI offline-safe e veloce.
- Aggiungere security regression tests dopo ogni finding del pentester.

Il sistema cognitivo è ben testato; quello che manca è il **perimetro** (CLI, API, MCP, sandbox). Una volta chiuso quello, HippoAgent è ready for v1.0.
